# Third Man — Cricket Scorer

A ball-by-ball cricket scoring PWA: toss, powerplay overs, dismissal tracking,
super overs, run-rate charts, PDF export, shareable scorecards, and a
Firebase-backed match history and leaderboard.

It's a plain static site — no build step, no framework, no bundler. `index.html`
loads `js/app.js` as a classic script and `js/firebase-init.js` as an ES module.

## Project structure

```
index.html              entry point
privacy.html            privacy policy (linked from the login screen; required by app stores)
css/styles.css          all styling
js/app.js               app state, screens, and scoring logic (single script)
js/firebase-init.js     Firebase app/auth/firestore setup (ES module)
icons/, manifest.json   PWA metadata and icons
firestore.rules         security rules for per-user match history
firebase.json           Firestore rules deploy config
_headers                response headers for static hosting (COOP for auth popups)
test/                   node:test suite for the scoring logic in js/app.js
capacitor.config.json   Capacitor config for the Android/iOS app wrappers
scripts/build-capacitor-www.js   copies the static site into www/ for Capacitor (see below)
android/, ios/          native app projects generated by Capacitor
```

## Running locally

Since it's a static site, any local web server works — opening `index.html`
directly from disk won't work because Firebase Auth popups require an http(s)
origin. For example:

```
npx serve .
# or
python -m http.server 8000
```

Then open the printed URL in a browser.

## Firebase

Auth (Google sign-in + anonymous/guest) and Firestore (per-user match history)
are configured in `js/firebase-init.js` against the `third-man-cricket`
Firebase project. Firestore security rules live in `firestore.rules` — deploy
changes to them with:

```
firebase deploy --only firestore:rules
```

The static site itself can be deployed to any static host (Cloudflare Pages,
Netlify, Firebase Hosting, GitHub Pages, ...); `_headers` sets the
Cross-Origin-Opener-Policy needed for the Google sign-in popup to work.

## Development scripts

```
npm test           # run the scoring-logic test suite (node's built-in test runner)
npm run lint       # eslint over js/ and test/
npm run format     # prettier --write over js/, css/, and html
npm run format:check
```

`npm install` is only needed for `lint`/`format` (eslint + prettier as dev
dependencies) — `npm test` uses Node's built-in test runner and has no
dependencies of its own.

Note: `js/app.js` has never been run through Prettier, so the first
`npm run format` will produce a large, purely cosmetic diff. Do that as its
own deliberate commit rather than mixed in with a logic change.

## Testing approach

`js/app.js` is a browser script, not a module, so `test/helpers/domStub.js`
stubs just enough of `document`/`window`/`localStorage` for it to load in
Node, and a small guarded `module.exports` block at the bottom of `app.js`
(a no-op in the browser) exposes the scoring functions to tests. See
`test/*.test.js` for coverage of ball-by-ball scoring, dismissals, innings-end
conditions, and a few regression tests for bugs found in past audits.

## Mobile apps (Android / iOS)

The site is wrapped for both app stores using [Capacitor](https://capacitorjs.com/),
which bundles the same static files into a native app shell rather than
loading them from a live URL — no service worker or separate offline story
needed, the UI ships inside the app binary.

`index.html`/`css/`/`js/`/`icons/`/`manifest.json`/`privacy.html` at the repo
root stay the single source of truth for the hosted web version. Nothing
about them changes for the native build; `npm run cap:build` (see
`scripts/build-capacitor-www.js`) just copies them into a generated `www/`
folder (gitignored), which is what Capacitor actually bundles.

```
npm run cap:build   # copy the static site into www/
npm run cap:sync    # cap:build, then npx cap sync (updates android/ and ios/)
```

Run `npm run cap:sync` any time you change `js/`, `css/`, `icons/`,
`index.html`, or add a Capacitor plugin.

### Why Google Sign-In needed a native plugin

Google blocks its OAuth flow inside plain embedded WebViews. The hosted web
version's `signInWithPopup`/`linkWithPopup` (in `js/app.js`) keeps working
unchanged for the website itself, but a bare WebView wrapper on iOS would
silently break Google sign-in. `js/app.js` now checks
`Capacitor.isNativePlatform()` (see `isNativePlatform()`,
`nativeGoogleCredential()`) and, on native builds, gets a Google credential
through the `@capacitor-firebase/authentication` plugin's real native
Sign-In UI instead of a WebView popup, then hands that credential to the
same Firebase JS SDK session (`window.__fb.auth`) the rest of the app
already uses — `onAuthStateChanged` in `js/firebase-init.js` doesn't need to
know which path was used. `continueAsGuest()` is untouched, since anonymous
sign-in is a plain network call with no popup involved on either platform.

### One-time Firebase console setup (manual — can't be automated)

The native Google Sign-In plugin needs the app registered in the Firebase
console, separately from the existing web app registration:

1. Firebase console → Project settings → Add app → Android. Package name
   must match `capacitor.config.json`'s `appId`
   (`com.thirdman.cricketscorer`) — change it there first if you want a
   different one; it's effectively permanent once published. Add your
   release keystore's SHA-1 (and SHA-256) fingerprint —
   `keytool -list -v -keystore your.keystore` — Google Sign-In won't work
   without this.
2. Download the resulting `google-services.json` into `android/app/`.
3. Add the Google Services Gradle plugin: `classpath
   'com.google.gms:google-services:4.4.2'` in `android/build.gradle`'s
   `dependencies`, and `apply plugin: 'com.google.gms.google-services'` at
   the bottom of `android/app/build.gradle`. (Left out of the repo for now
   — applying it without a real `google-services.json` present breaks the
   build.)
4. Firebase console → Add app → iOS, using the same bundle ID you set up in
   Xcode. Download `GoogleService-Info.plist` into `ios/App/App/` and add
   it to the Xcode project. Add the reversed client ID URL scheme it
   specifies to the app's Info.plist (`CFBundleURLTypes`) — the Firebase
   console's setup instructions show the exact value for your project.

### Android build

Needs a JDK and the Android SDK (simplest: install Android Studio, which
includes both). From this repo:

```
npm run cap:sync
cd android
./gradlew assembleDebug      # unsigned debug APK, for local testing
```

Or open the `android/` folder directly in Android Studio and run/debug from
there — easier for testing on an emulator or a plugged-in device.

For a Play Store release build you'll need a signing keystore
(`keytool -genkey -v -keystore release.keystore -keyalg RSA -keysize 2048
-validity 10000 -alias thirdman`, kept out of git) wired into
`android/app/build.gradle`'s `signingConfigs`, then `./gradlew
bundleRelease` to produce the `.aab` Play Console wants.

### iOS build

Needs Xcode, which only runs on macOS — not possible from this environment,
so `ios/` is fully scaffolded and synced but unbuilt. Once you have Mac or
cloud-CI (e.g. Codemagic, GitHub Actions macOS runners) access:

```
npm run cap:sync
npx cap open ios   # opens ios/App/App.xcworkspace in Xcode
```

From Xcode: set your Apple Developer team for signing, add
`GoogleService-Info.plist` (see above), then Product → Archive to produce a
build for TestFlight/App Store Connect. The Mac is only needed to build and
sign — the resulting app runs on any iPhone like any other App Store app.

### Publishing (both stores need your own account — not something I can do for you)

- **Google Play Console** (`https://play.google.com/console`, $25
  one-time): create the account, create an app listing, fill out the Data
  Safety form (what's collected is documented in `privacy.html` — account
  info via Google Sign-In, match summaries), upload the signed `.aab` to an
  internal testing track first, then promote to production once it's
  working.
- **Apple Developer Program** (`https://developer.apple.com/programs/`,
  $99/year): create the account, create the app in App Store Connect,
  upload a TestFlight build first, fill out the App Privacy "nutrition
  label" (same data as above), then submit for review. Bare "website in a
  wrapper" apps risk rejection under App Review Guideline 4.2 (Minimum
  Functionality) — the native shell plus native sign-in here meaningfully
  helps, but a polished, app-like feel on first launch still matters for
  getting through review.
- Both stores require the privacy policy URL — use wherever `privacy.html`
  ends up hosted alongside the rest of the site.
